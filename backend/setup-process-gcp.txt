Phase 1: Provisioning the Free GCP Instance
Machine Selection: Created an e2-micro instance (2 vCPUs, 1 GB RAM).

Location: Set the region to us-central1-a (required for the Always Free tier).

Storage: Changed the boot disk to a 30 GB Standard Persistent Disk running Debian (avoided the default "Balanced" disk to prevent charges).

Networking: Enabled "Allow HTTP" and "Allow HTTPS" traffic, and changed the Network Service Tier to Standard to avoid premium routing costs.

RAM Optimization: Disabled the "Ops Agent" in the Observability tab to free up ~150MB of RAM.

Cost Prevention: Disabled Snapshot schedules in the Data Protection tab to prevent automated backup charges.

Phase 2: OS Stabilization (The Swap File)
Since the e2-micro only has 1 GB of RAM, we created "virtual RAM" on the hard drive to prevent Out-Of-Memory (OOM) crashes when running Node.js and Docker simultaneously.

System Update: Cleared the locked background package manager and ran sudo apt update && sudo apt upgrade -y.

Create Swap: Allocated a 2 GB file using sudo fallocate -l 2G /swapfile.

Secure & Format: Set strict permissions (sudo chmod 600) and formatted the file (sudo mkswap).

Activate: Turned on the swap space (sudo swapon) and made it permanent by adding it to /etc/fstab.

Phase 3: Installing the Tech Stack
Node.js & npm: Added the NodeSource repository and installed Node.js v20.x and the npm package manager.

Docker Installation: Installed the Docker engine (sudo apt install docker.io -y) to handle the isolated code execution environments.

User Permissions: Added the current user to the Docker group (sudo usermod -aG docker $USER and newgrp docker) to allow running containers without needing sudo.

Environment Prep: Manually pulled the python:3.10-alpine Docker image so it wouldn't cause a timeout during the first code execution.

Phase 4: Building the CodeScript Execution Engine
Project Initialization: Created the codescript-backend folder and initialized a Node project (npm init -y).

Docker Service (dockerService.js): Wrote the core execution logic using Node's child_process.

Security: The script spawns containers with --network none (no internet access), --memory="128m" (RAM limit), and --cpus="0.5" (CPU limit).

Timeout Tuning: Increased the execution timeout to 15 seconds to account for the free server's hardware latency when booting up containers.

Phase 5: Remote Development Environment (VS Code SSH)
To move away from browser-based terminals, we established a direct, secure connection between a local IDE and the cloud VM.

SSH Key Generation: Generated an RSA 3072 SSH key pair locally using PowerShell (ssh-keygen) to avoid circular authentication issues on the server.

GCP Metadata Integration: Injected the public key into the GCP Compute Engine instance metadata, ensuring the key was specifically bound to the existing smsram001 user profile to preserve Docker permissions.

VS Code Configuration: Updated the local .ssh/config file with the VM's External IP and the designated IdentityFile.

RAM Protection: Configured VS Code settings to explicitly exclude **/node_modules/* and **/temp/* from "File Watching" to prevent the IDE's background processes from consuming the e2-micro's limited memory.

Phase 6: API Architecture & Database Integration (TiDB + Prisma)
We transformed the isolated execution script into a structured, MVC-patterned backend and connected it to a remote database.

Project Scaffolding: Executed a single-line bash command to instantly generate a production-ready folder structure (src/controllers, routes, middleware, services, sockets).

Runtime Upgrade (NVM): Installed Node Version Manager (NVM) to seamlessly upgrade the environment to Node.js v20 (LTS), resolving strict engine requirements for modern backend libraries.

Database ORM Setup: Installed Prisma v5 (intentionally bypassing v7 to avoid experimental driver adapter complexities) and configured schema.prisma with a MySQL provider to connect to the remote TiDB instance.

Data Modeling: Defined the relational schema for User, Contest, Problem, and Submission tables, including Enums for Role-Based Access Control (STUDENT, WRITER, ADMIN).

Database Synchronization: Ran npx prisma db push to push the schema directly to TiDB, followed by npx prisma generate to build the local TypeScript/JavaScript client.

Seeding the Admin: Wrote a seed.js script utilizing bcryptjs to securely hash a password and inject the inaugural "Super Admin" account into the live database.

Phase 7: Production Web Server & SSL (Nginx + DuckDNS)
We upgraded the Node.js app from a raw IP on Port 3000 to a fully secure, production-grade HTTPS domain.

Domain Registration: Claimed a free dynamic domain via DuckDNS and mapped it to the GCP VM's External IP address.

Reverse Proxy (Nginx): Installed Nginx to intercept standard web traffic (Ports 80/443). Configured the proxy_pass block to route requests to the internal Express server on localhost:3000.

WebSocket Preparation: Explicitly injected Upgrade and Connection headers into the Nginx config to ensure future Socket.io real-time traffic (for live exam monitoring) wouldn't be blocked.

SSL Encryption (Certbot): Installed Certbot and the Nginx plugin to automatically fetch and configure a free Let's Encrypt SSL certificate, enforcing HTTPS redirection.

Firewall Configuration: Used the GCP Console (VPC Network) to create an ingress firewall rule allowing tcp:80 and tcp:443 traffic from 0.0.0.0/0.

Process Management (PM2): Installed PM2 globally via npm to daemonize the Express server (pm2 start src/server.js). Ran pm2 save to ensure the API automatically resurrects if the Google Cloud server ever reboots or crashes.
